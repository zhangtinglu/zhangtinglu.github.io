归档库选择-mysql vs tidb vs clickhouse vs ob

发表于 2021-12-23 更新于 2022-01-06

背景

我们希望将MySQL里面的冷数据归档出来，那就需要选择一个存储引擎来承接这部分数据。因此我们对目前市面上主流的存储进行对比：mysql innodb、mysql tokudb、tidb、ob、clickhouse

测试说明

物理机器

机器 OS CPU 内存 磁盘
Host1 CentOS Linux release 7.5.1804 (Core) 40 Intel(R) Xeon(R) Silver 4114 CPU @ 2.20GHz 187G nvme0n1 1.5T
Host2 CentOS Linux release 7.5.1804 (Core) 48 Intel(R) Xeon(R) Silver 4214 CPU @ 2.20GHz 125G nvme0n1 1.5T
Host3 CentOS Linux release 7.5.1804 (Core) 32 Intel(R) Xeon(R) CPU E5-2620 v4 @ 2.10GHz 125G nvme0n1 5.8T

测试说明

内存配置：所有节点同意配置4GB

CPU核数：4核

写入数据：5个表，每个表5百万行数据

测试结论

1、写入物理盘

存储 写入时间 存储量 修改字段/添加字段 读写(QPS/95)
mysql innodb 113S 5.8G 50.10S/23.57S 12250.11 / 99.33
mysql tokudb 425S 3.1G 0.02S/0.03S 9814.30 / 112.67
ob 907S 7G 0.042/0.030S 26290.06 / 30.81
tidb 10001S 5.4G 0.06S/0.23S 15418.84 / 47.47

2、写入共享存储chubaoFS

存储 写入时间 存储量 修改字段/添加字段 读写(QPS/95)
mysql innodb 748S 5.8G 18 min 10.95S/14 min 42.88 sec 2992.73 / 1013.60
mysql tokudb 513S 3.1G 0.13S/0.11S 9394.43 / 102.97
ob 1232S 7G 0.090S/0.042S 1702.64 / 580.02
tidb 1300S 5.4G 0.55S/1.98S 6690.92 / 142.39

总结：

最适合搭配共享存储chubaofs的来做归档的是tokudb引擎，原因如下：

1. 对比innodb，在完全顺序写入的情况下，比innodb节省约50%的空间，在有数据空洞的情况下，压缩率能达到2~10倍不等。
1. 对比ob，ob内存至少8G才能有不错的表现，而且ob在共享存储上的读写性能降低严重，不适合用在归档的场景中，存储压缩率对比innodb也并无优势。ob在大数据量&高并发场景中会有更亮眼的表现。
1. 对比tidb，同等条件下，tidb的无论是内存使用还是性能都比不过ob。
1. 对比clickhouse，clickhouse并发大概在200左右，只能保证最终一致性，对于update、delete一类的操作不友好，不适合实时online的查询。
1. 共享存储，对比所有上述存储，tokudb在共享存储上的表现最好，对比物理存储的性能并没有太大的损失。

tokudb优势

​ 比较高的压缩比例，由于innodb存储2~10倍不等。

​ 秒级别的改表，加减字段

​ 支持ACID

tokudb缺点

​ 不支持外键

​ text、blob字段改表不行

​ CPU消耗相对高

ob对内存有很大的需求，内存大小对性能影响非常大，4G这样的内存，测试使用内存10G – 如果使用64G，性能能提升10倍。对每一张表做计算存储分离不现实

共享存储随机写性能很差

clickhouse提供的是最终一致性，要提供线上一致性还是需要
